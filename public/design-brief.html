Internal positioning brief

Ordino — QA for the AI era

Coding agents ship code faster than QA can keep up. Ordino gives the person responsible for product quality — a QA engineer, tech lead, or release manager — agentic tools for the work they already do: test planning, authoring, automation, root cause analysis, and flaky test management.

Who Ordino is for

The quality owner, not the developer

Ordino is built for whoever is accountable for product quality — a QA engineer, a tech lead, or a release manager. We call this person the quality owner throughout this document. Ordino makes the tasks that person already does today faster and more reliable; it isn't a tool that replaces them, and it isn't primarily a developer tool.

The problem

AI is writing code faster than QA can verify it. The average developer now submits 7,839 lines of code per month, up from 4,450 a year ago — a 76% increase — and over 85% of enterprise QA teams report that AI code generation has created a testing bottleneck they can't keep pace with. The agentic era needs agentic methods of quality assurance, not more manual QA headcount.

For a quality owner working across a product built from many services and repos, that bottleneck shows up as blind spots, not just backlog: when a developer changes one piece, nothing today tells the quality owner what else it breaks — until CI fails somewhere else, hours or days later, often after they've already signed off on the change.

Practically, that's a prioritization problem, not a coverage problem. A quality owner can't run the full suite for every change and still keep pace with how fast that change ships — and today, nothing tells them with confidence which tests actually matter for a given change and which don't. Research bears this out: 60% of organizations already ship untested code, citing sheer volume as the leading reason — not carelessness, but not knowing where to focus. In interviews with three developers, that same gap showed up concretely: the cross-repo blind spots above meant nobody could say which tests actually covered a change until it had already shipped.

"I don't have time to run everything, and I don't have a reliable way to know which tests actually matter for this change." — synthesized from developer interviews, reframed to the quality owner's seat

This single constraint — knowing what to test, before the PR exists — rules out most of the current market. CI checks, PR bots, and post-merge coverage tools all live too late to solve it. This is the hero problem the rest of the platform is built around.

The positioning bet

What we are

The quality owner's own QA tool

Ordino is where the quality owner does their work directly — chat, terminal, or an IDE panel — and separately, agent-agnostic where it counts: any coding agent a team already uses (Claude Code today, others tomorrow) can call Ordino via MCP without Ordino trying to live inside that agent's own loop.

What we are not

A tool that asks developers to change their loop

Ordino doesn't compete for the developer's attention or ask them to adopt a new interactive front-end — their coding agent stays exactly as it is, and calls Ordino via MCP when it needs to. It's the quality owner, not the developer, who has a standing reason to work inside Ordino directly.

Why this is defensible, not just a feature

Coding agents can already be prompted to "write some tests." What they can't do is see across repos, connect to requirement, test-management, and communication tools, or persist context between sessions — that's structurally outside a single agent's scope. Ordino's edge isn't a better prompt; it's standing awareness of the whole system, not one working directory.

How it works: three surfaces, one loop

The quality owner's primary interface is Ordino itself. Two more surfaces operate around that, without ever requiring the quality owner to switch tools or the developer to change their loop:

This resolves the core tension: the proactive, ambient QA value from the original vision doesn't require developers to adopt a new primary tool — it just requires Ordino to notify through channels they already trust, and to be callable by whatever agent a developer already uses.

A key architectural honesty check

Today, Ordino is a tool, not an agent. The coding agent calls it via MCP to perform front-end test automation — a deterministic, callable function. There is no "Ordino Agent" yet. Building one (or several) is the next architectural leap, not an incremental feature.

That leap needs one guardrail stated up front: an agent calling another agent introduces compounding nondeterminism. To keep Ordino trustworthy as a QA product, its facts — coverage numbers, pass/fail results, risk verdicts — should come from deterministic verification (actually running tests, actually computing coverage, actually tracing dependencies), with agentic reasoning used only to prioritize, explain, and act on top of those facts — never to invent them.

Rather than jumping straight to the hardest agent (cross-repo risk), the plan is to prove the "Ordino Agent" architecture on the smallest, best-understood domain first: turn the existing front-end test automation tool into a risk-aware test automation agent — one that decides what to test based on what changed, not just what it's told to test. That's the same reasoning the hero capability needs (assess risk, then act), scoped to one repo instead of many. It de-risks the architecture before the harder cross-repo bet, and it's a direct stepping stone rather than a detour.

Architecture risks and mitigations

Raised by our architect: a coding agent calling an Ordino agent introduces real risks. Each one points to a specific design choice, not a reason to abandon the bet.

Risk: the calling agent forgets or mangles context, causing mistakes

Don't trust the messenger — verify against ground truth. Ordino already has direct access to the local repo and git history, so the MCP tool call should be a thin trigger ("something changed, here's the repo/commit"), with Ordino independently reading the actual diff and files itself rather than relying on whatever summary the calling agent passes along.

Risk: no control over the other agent's context (AGENTS.md conflicts, compaction, unknown harness-level ingestion)

This is the strongest argument for weighting the background watcher over the on-demand tool call for anything high-stakes. The on-demand MCP path is fine for convenience-tier tasks, but it shouldn't be the channel of record for judgment calls like cross-repo risk. The independent daemon runs its own process with its own context that Ordino fully controls — it doesn't depend on what's happening inside a harness we can't inspect.

Risk: our output gets ingested into a model provider's training pipeline, eroding our edge over time

No clean technical fix, and worth being honest about that. Reasoning patterns (how to think about cross-repo risk) will likely diffuse into frontier models over time regardless — that shouldn't be the moat we defend. What can't transfer through training is the standing, continuously-updated, customer-specific state: a live dependency graph tied to one org's actual repos, requirements, and history. Practically: push customers toward zero-data-retention/no-training API tiers where available, and keep Ordino's outputs concise and structured rather than exposing verbose reasoning through the host agent's visible transcript.

Who this is for

The pain is sharpest for engineering orgs where the product is split across many repos or services — typically mid-size to large teams, often ones running a microservices-style architecture, where no single developer can hold the whole dependency graph in their head. This is a team/platform-level need, not an individual-developer nice-to-have. Concretely, it's the QA engineer, tech lead, or release manager at these orgs — whoever is accountable for quality — that Ordino is built for.

Open source & monetization strategy

An open-core split: open source the parts that build trust and reach, keep proprietary the parts that are the actual moat.

Open source

CLI, MCP tool shell & connectors

The plumbing that lets any coding agent call Ordino, and the individual connectors to git providers, Jira, and Slack. Being inspectable lowers the trust bar for letting something read a team's code, requirements, and chatter — and lets the community add connectors we haven't built yet.

Proprietary / hosted

Cross-repo knowledge engine & risk reasoning

The continuously-running, ingested-and-indexed graph across repos, the risk-assessment reasoning on top of it, team-scale notification intelligence, and enterprise controls (permissions, audit, SSO). This is inherently a hosted service, and it's the actual differentiator — so it's what we charge for.

Pricing shape

Banded repo tiers (e.g. up to 10 / up to 50 repos), not linear per-repo pricing — repo count is the real value driver, since blast-radius awareness scales with how much of the system Ordino can see, but taxing every repo connected would discourage the very behavior (connect everything) the product needs to work well. Seats bundled generously within a plan rather than metered tightly — seat count isn't the real value driver; a 5-person team across 40 services needs Ordino more than 50 people in one repo. Usage-based overage on the compute-heavy agentic actions (test runs generated, PRs risk-assessed) beyond what's bundled, since that cost genuinely scales with how hard the agents are working.

Buyer motion: organizations are the primary target — a platform/DevEx lead buying a banded plan for the team — but individual developers aren't locked out. An individual can adopt the open-source CLI and connectors solo, become the internal champion, and pull their org into the paid hosted tier. Bottom-up adoption feeding a top-down sale.

Data residency & local execution (future)

A future claim worth being precise about: "nothing leaves the organization's environment" only holds if there's no frontier-LLM escalation. Split by layer instead of making one blanket claim — the deterministic layer (test execution, coverage computation, dependency tracing) isn't an LLM operation at all, local or cloud, so it can genuinely never send code anywhere for any customer. The LLM/reasoning layer is where a local-vs-frontier tradeoff exists, and it's weakest exactly where the hero capability needs it most.

Open questions for the team

  1. What is an "Ordino Agent," architecturally? One orchestrating agent with several skills, or several independent specialized agents (coverage, requirements, notifications)? This shapes everything downstream.
  2. Technical feasibility of cross-repo risk. Pre-PR blast-radius detection requires a live, cross-repo dependency picture that stays in sync with uncommitted local changes — not a one-off scan. What's the right approach, and what's a realistic v1 scope?
  3. Platform risk. If Anthropic, Cursor, or others build native cross-repo awareness into their agents, our agent-agnostic bet weakens. What's our leading indicator for when to accelerate from Phase 1 to Phase 2/3?
  4. Scope discipline. With several capabilities now on the table, how do we avoid building all of them shallowly instead of the hero (cross-repo risk) deeply?

Riskiest assumption

That developers/orgs will value cross-repo, pre-PR QA awareness enough to adopt Ordino as standing infrastructure — rather than treating today's wedge (agent-callable test automation) as the whole product. Three interviews is a strong signal, not proof at scale. A secondary risk: spreading effort across many capabilities before the hero capability is proven could dilute the thing that actually differentiates Ordino. A third, newer risk from the quality-owner pivot itself: that it's the quality owner, not the developer, who adopts Ordino as their own standing tool — unvalidated directly, and a different buyer/user motion than the original developer-first framing assumed.
